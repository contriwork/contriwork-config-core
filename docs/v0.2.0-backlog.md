## Issue 1 (H) — README uvicorn warning + thread-fallback bootstrap

**Title:**

```
docs(README): warn about uvicorn --reload subprocess + threading bootstrap fallback
```

**Labels:** `enhancement` `v0.2.0` `documentation`

**Body:**

````markdown
## Context

During the Digital Worker (FastAPI + Python 3.13) integration of v0.1.0, the naive bootstrap pattern — calling `await load_config(...)` inside an `async def` lifespan handler, with a sync wrapper at module-import time — failed in a non-obvious way under `uvicorn --reload`.

uvicorn's reload subprocess imports the user app (`main:app`) **from inside an already-running uvloop event loop**. Any module-import-time call that wraps `asyncio.run(load_config(...))` raises:

```
RuntimeError: asyncio.run() cannot be called from a running event loop
```

This is reachable in any project that has module-level code reading a config value (e.g. `BASE_URL = get_config().coingecko_base_url` as a class attribute). The error is silent in the sense that the bootstrap is skipped, the singleton stays `None`, and the failure surfaces later as a `RuntimeError` in unrelated code.

## Problem

The current README does not mention this. A new user adopting the package in a hot-reload-capable web framework (FastAPI/Starlette/Quart with uvicorn reload, hypercorn reload, etc.) will hit it on first run and spend an hour diagnosing it.

The naive try/except `get_running_loop()` workaround **does not work** — it correctly detects the running loop but then has no fallback path:

```python
# DOES NOT WORK
try:
    asyncio.get_running_loop()
    return  # _CONFIG stays None → module-level access raises later
except RuntimeError:
    pass
_CONFIG = asyncio.run(load_config(...))
```

## Proposal

Add a section to README.md (suggested heading: `### Bootstrapping inside an async server`) that documents:

1. **The trap**: brief explanation of the uvicorn-reload-subprocess issue.
2. **The working pattern**: isolate the bootstrap to a separate thread with its own event loop. Verbatim code that worked in production:

```python
def _load_blocking():
    """Load config synchronously, even when called from inside a running loop.

    Why: uvicorn --reload imports the app from inside its own uvloop event
    loop. asyncio.run() refuses to nest, so we delegate to a fresh thread.
    """
    import asyncio
    import threading

    try:
        asyncio.get_running_loop()
        has_running_loop = True
    except RuntimeError:
        has_running_loop = False

    if not has_running_loop:
        return asyncio.run(load_config(...))  # normal path

    result, error = [], []

    def _runner():
        try:
            result.append(asyncio.run(load_config(...)))
        except BaseException as e:
            error.append(e)

    t = threading.Thread(target=_runner, daemon=True)
    t.start()
    t.join()

    if error:
        raise error[0]
    return result[0]
```

3. **Lazy-fallback note**: the synchronous `get_config()` accessor should call the same pattern so module-load failures degrade gracefully (rather than surfacing as a confusing `RuntimeError` from unrelated code).

## Acceptance

- [ ] README has a "Bootstrapping inside an async server" section near the Quick Start.
- [ ] The threading-fallback pattern is included as runnable Python with a one-paragraph explanation of the trap.
- [ ] Cross-referenced from the lifespan example (if any) and from `CONTRACT.md`'s idempotency / re-entrancy section.

## Out of scope

- Changing `load_config`'s actual signature or making it sync. The issue is documentation-only; the async API is correct, and the fix lives in the caller.
- Embedding the helper in the package itself (it's caller-policy code — every app's bootstrap is slightly different, and we don't want to ship a "framework"-feeling utility).

## References

- Real-world hit: Digital Worker integration, PR 4 — see `PACKAGES_ROADMAP.md` §10 "config-core v0.1.0 done — bulgular & v0.2.0 kapsamı" block.
````

## Issue 2 (H) — `EnvSource` opt-in JSON decode for list/dict

**Title:**

```
feat(EnvSource): opt-in JSON decode for list/dict env vars
```

**Labels:** `enhancement` `v0.2.0`

**Body:**

````markdown
## Context

`EnvSource` returns every value as a raw string, per CONTRACT v1's "coercion is the schema adapter's job" rule. This is correct in principle — the source shouldn't make schema assumptions — but in practice every Python project that has a `list[str]` or `dict[str, X]` config field ends up writing a `BeforeValidator(json.loads)` (or equivalent CSV parser) on the schema side.

The Digital Worker integration kept its existing `_csv_to_list` BeforeValidator because operators were already comfortable with `KEY=a,b,c` in `.env` files, and CSV is friendlier than `KEY=["a","b","c"]` for shell pasting. But the project also has `dict`-typed fields (e.g., `RATE_LIMITS={"market_data": 10}`) and there a JSON literal is the only sane format — so we still ended up with a `BeforeValidator(json.loads)` for those.

## Problem

Every consumer reinvents the same string-to-`list`/`dict` coercion. This is a DX papercut: 5 lines of boilerplate per typed-collection field. `pydantic-settings` addresses this with `env_parse_enums` + JSON-decode-on-list/dict; we should match that ergonomic without giving up the "schema owns coercion" principle.

## Proposal

Add an opt-in `decode_json_for` parameter to `EnvSource`:

```python
EnvSource(
    prefix="APP_",
    decode_json_for=("list", "dict"),  # default: () → no decoding (current behavior)
)
```

When set, `EnvSource.snapshot()` attempts to `json.loads(value)` for keys whose schema-declared type matches one of the named categories. The set of categories (suggested: `"list"`, `"dict"`, `"bool"`, `"int"`, `"float"`) covers the common cases without expanding into "JSON everywhere".

Because the source doesn't know the schema, the decode is **best-effort**: if `json.loads` fails, the raw string is passed through (the schema will either accept it or raise `VALIDATION_FAILED` as today). This avoids the source becoming a hard error gate.

Alternative considered: a separate `JsonEnvSource` adapter. Rejected because 99% of projects want most fields as plain strings and only a handful as JSON; forcing two sources is heavier than a single opt-in flag.

## Acceptance

- [ ] `EnvSource(decode_json_for=...)` parameter added; default `()` preserves v0.1.0 behavior.
- [ ] When the env value parses as JSON of the requested category, the parsed value is returned in the snapshot dict (nested dicts merge correctly).
- [ ] When the value fails to parse, the raw string is returned (current behavior); schema validator decides.
- [ ] Contract test added under `contract-tests/test_cases.json`.
- [ ] Three-language parity: `.NET` and `npm` adapters expose equivalent flag (`DecodeJsonFor` / `decodeJsonFor`).

## Notes

- Does **not** bump CONTRACT revision — `EnvSource` is allowed to grow optional parameters that default to current behavior (CONTRACT.md "Invariants").

## References

- pydantic-settings JSON decode reference: `BaseSettings.model_config['env_parse_enums']`
- Real-world hit: Digital Worker integration retained `_csv_to_list` BeforeValidator because no native option exists.
````

## Issue 3 (H) — `FileSource` native dotenv format

**Title:**

```
feat(FileSource): native dotenv format support
```

**Labels:** `enhancement` `v0.2.0`

**Body:**

````markdown
## Context

The Python ecosystem standardized on `.env` files for local dev and for many container-deployment patterns (Docker Compose `env_file`, Kubernetes `envFrom`, dotenv-cli for CI). v0.1.0's `FileSource` supports `yaml`, `json`, and `toml` only.

The Digital Worker integration kept `python-dotenv` as a separate dependency purely for `.env` reading because moving to `local.yaml` would have broken operator muscle memory and the existing `.env.example` documentation.

## Problem

Every Python consumer adopting config-core in a project that already has `.env` either:

1. Migrates to YAML/JSON/TOML (operator-facing breakage; rejects bringing the package in).
2. Keeps `python-dotenv` and pre-loads it into `os.environ` before `EnvSource` sees it (works, but adds a dependency the package was supposed to eliminate, and feels duplicative).

## Proposal

Extend `FileSource` to accept `format="dotenv"`:

```python
FileSource("./.env", format="dotenv")
```

Behavior:

- Read the file line-by-line, ignoring comments (`#`) and blank lines.
- Parse `KEY=VALUE` (the dotenv subset that 95% of users actually use).
- Expand the value's quoting (`KEY="value with spaces"` → `value with spaces`).
- Return the result as a flat snapshot dict (no nesting; dotenv is flat).
- File extension `.env` infers `format="dotenv"` automatically (so the explicit `format=` is optional).

Explicitly **not** in scope (these are dotenv-spec corners that diverge across implementations):

- Variable interpolation (`KEY=$OTHER`). Use a `SecretResolver` for that.
- Multi-line values with backticks.
- `export KEY=...` prefix (silently strip if present, but don't advertise).

The implementation can be ~30 lines of regex parsing without a dependency on `python-dotenv`, which keeps the package's "no I/O dependencies beyond stdlib + the user's chosen format libs" posture.

## Acceptance

- [ ] `FileSource(path, format="dotenv")` works.
- [ ] `.env` extension inferred without explicit `format=`.
- [ ] Contract test fixtures cover: comments, blank lines, quoted values, empty values, equals-sign-in-value.
- [ ] Three-language parity: same fixture parses identically in Py / .NET / TS.
- [ ] README "Source formats" table updated.

## Notes

- Does **not** bump CONTRACT revision (additive parameter on an existing source).
- The `FileSource` `format` enum becomes `{"yaml", "json", "toml", "dotenv"}`.

## References

- Real-world hit: Digital Worker integration retained `python-dotenv` purely because the package didn't read `.env` natively.
````

## Issue 4 (M) — `load_config` async signature visible to inspect

**Title:**

```
fix(types): load_config async signature visible to inspect.signature
```

**Labels:** `bug` `v0.2.0` `types`

**Body:**

````markdown
## Context

`load_config` is an async function — the caller must `await` it. CONTRACT.md documents this clearly. However, when the signature is introspected via `inspect.signature(load_config)` or by an IDE that surfaces the call hint, the function appears synchronous: no `Awaitable[T]` / `Coroutine[Any, Any, T]` wrapper on the return type, no `async def` annotation visible.

This caused a real first-attempt error in the Digital Worker integration: the integrator wrote `cfg = load_config(...)` (no await), got a `Coroutine` object back, and only realized via runtime `TypeError` that an `await` was needed.

## Problem

Static type checkers (`mypy`, `pyright`) catch this in the user's code, but only when the user has type hints active and is running the checker. Many projects' `__main__` scaffolding or quick smoke scripts bypass the checker. Even with type hints, the IDE hover tooltip shows the misleading signature first, then the user has to read CONTRACT.md to learn it's async.

The likely cause is a `@functools.wraps` or similar decorator that strips the coroutine marker from the underlying function. Or a `typing.Protocol` pattern where the protocol's `def load_config(...)` is sync-shaped while the concrete implementation is `async def`.

## Proposal

Audit the `load_config` definition and decorators in `python/src/`. Ensure that:

1. `inspect.iscoroutinefunction(load_config)` returns `True`.
2. `inspect.signature(load_config)` shows the return type as `Coroutine[Any, Any, T]` (or `Awaitable[T]` in the public type stub).
3. IDE hover (PyCharm / VS Code Pylance) shows the function as async.

If the public surface is a `typing.Protocol`, ensure the protocol method is declared as `async def` so structural subtyping reflects the async-ness.

## Acceptance

- [ ] `inspect.iscoroutinefunction(load_config)` is `True` after import from the public `contriwork_config_core` namespace.
- [ ] Type stub (`.pyi` if present, otherwise inline annotations) declares the return type with the `Awaitable` wrapper.
- [ ] IDE smoke test: hover in VS Code (Pylance) shows `(schema, sources, resolver=None) -> Coroutine[Any, Any, T]`.
- [ ] Repeat for .NET (`Task<T>` visible) and TypeScript (`Promise<T>` visible) — these are likely already correct but verify.

## Notes

- Pure type-system fix; no runtime behavior change.
- Does **not** bump CONTRACT revision.

## References

- Real-world hit: Digital Worker integration first-attempt error.
````

## Issue 5 (M) — `secret_str_or_empty` utility

**Title:**

```
feat: secret_str_or_empty utility for SecretStr unwrapping
```

**Labels:** `enhancement` `v0.2.0`

**Body:**

````markdown
## Context

When using `pydantic.SecretStr` for sensitive config fields (`database_url`, `openai_api_key`, etc.), every read site looks like:

```python
db_url = (cfg.database_url.get_secret_value() if cfg.database_url else "")
```

The Digital Worker integration ended up with this pattern in 8+ places. It's easy to forget the `if … else ""` guard for optional fields, leading to `AttributeError: 'NoneType' object has no attribute 'get_secret_value'` at runtime.

## Problem

This is boilerplate that every consumer ends up writing. The package already hard-recommends `SecretStr` in CONTRACT.md and in error-taxonomy notes; it should also provide the safe-unwrap helper.

## Proposal

Add a tiny utility to the public namespace:

```python
def secret_str_or_empty(value: SecretStr | None) -> str:
    """Return the unwrapped secret string, or "" if the field is None.

    Centralizes the SecretStr unwrap pattern so callers don't repeat the
    None-guard in every read site.
    """
    return value.get_secret_value() if value is not None else ""
```

Optionally also add:

```python
def secret_str_required(value: SecretStr | None, field_name: str) -> str:
    """Same as above but raises if the field is None."""
    if value is None:
        raise ValueError(f"required secret field {field_name!r} is None")
    return value.get_secret_value()
```

Both exported from `contriwork_config_core` namespace. Three-language parity: `.NET` and `npm` adapters provide equivalent helpers (`SecretStrOrEmpty`, `secretStrOrEmpty`).

## Acceptance

- [ ] `secret_str_or_empty` and `secret_str_required` exported from the public namespace.
- [ ] Type signatures are precise (no `Any`).
- [ ] Unit tests cover: `None`, valid `SecretStr`, accidentally-passed plain `str` (should raise `TypeError` not silently coerce).
- [ ] README usage example added.

## Notes

- Does **not** bump CONTRACT revision (utility addition, not protocol change).
- Considered: making this a method on `SecretStr` itself. Rejected because we don't own pydantic's class.

## References

- Real-world hit: 8+ unwrap sites in Digital Worker integration.

## Issue 6 (L) — `load_config(resolver=None)` semantics

**Title:**

```
docs/code: clarify load_config(resolver=None) semantics
```

**Labels:** `documentation` `v0.2.0`

**Body:**

````markdown
## Context

CONTRACT.md §`load_config` parameters states:

> `resolver` — optional. If omitted, defaults to `EnvResolver()` (env-only secret resolution). Passing an explicit `None` / `null` disables secret resolution entirely (any `${…}` string passes through unchanged).

This is **two different defaults** for the same `None` value depending on whether the argument is "omitted" or "explicitly passed as `None`". In Python this distinction requires either a sentinel object (`_UNSET`) or `**kwargs` inspection — neither of which the casual reader of `def load_config(schema, sources, resolver=None)` will infer from the signature.

## Problem

The current Python signature is:

```python
async def load_config(schema, sources, resolver=None) -> T: ...
```

Reading this, every Python user assumes `resolver=None` and "omitted" are identical. The contract says they aren't. This is a footgun for anyone who writes `load_config(schema, sources, resolver=None)` expecting the default behavior — they actually disable secret resolution.

## Proposal

Pick one of these resolutions:

**Option A (preferred): match the signature to the contract.** Use a sentinel:

```python
_UNSET: Any = object()

async def load_config(schema, sources, resolver=_UNSET) -> T:
    if resolver is _UNSET:
        resolver = EnvResolver()  # default
    elif resolver is None:
        resolver = NullResolver()  # explicit opt-out
    ...
```

Add a `NullResolver` to the public namespace so the opt-out is named:

```python
load_config(schema, sources, resolver=NullResolver())  # crystal clear
```

This way the signature reads honestly and the explicit-opt-out path has a self-documenting name.

**Option B: collapse the two semantics.** Make `resolver=None` always mean "no resolution" (current explicit-opt-out behavior), and require callers who want `EnvResolver` to pass it explicitly. CONTRACT.md is updated.

This is breaking for v0.1.0 users who relied on the "omitted = EnvResolver" default, so it would need a deprecation cycle.

**Recommendation: Option A.** Non-breaking, matches the contract, and gives the opt-out path a name.

## Acceptance

- [ ] Decision made (A or B).
- [ ] If A: sentinel implementation lands; `NullResolver` exported; README and CONTRACT.md examples updated.
- [ ] If B: deprecation warning in v0.2.0, removal in v0.3.0; CONTRACT.md revised.
- [ ] Three-language parity: same decision applies in `.NET` and `npm`.

## Notes

- Option A does **not** bump CONTRACT revision (signature semantics preserved, only made explicit in code).
- Option B **does** bump CONTRACT revision (behavior change for the `resolver=None` path).

## References

- CONTRACT.md §`load_config` parameter table.
- Real-world note: spotted during Digital Worker integration review; not yet encountered as a runtime bug (DW always passes `EnvResolver` explicitly).
````
